Basic git commands

May 5



add

git add could be used in many different ways for example.

~> git add filename #This command adds individual file.
~> git add . #This command adds only those files which has been added and/or modified in the repo.
~> git add --all #This command can add preety much all types of files, added, modified and deleted.
~> git add -A # this is same as git add --all

This git add command will stage the file, which means now git knows about those changes we have made on the file. This change however has not permanently been recorded in the repo yet. Next step after git add would usually be git commit, which records the changes on git permanently. However, if you wish to not record those changes or reset it, git reset command will do that for you.

branch

git branch commands:-

~> git branch branchname #creates repo's branch
~> git branch -d branchname #deletes the branch

So when and why do you branch?

  • Real world developer do not work on their master branch, instead they create branch any feature they want to add on the app, and if that feature makes sense, they merge it over to master. If incase anything goes wrong, they can always git rid of that branch and not worry about the changes and correction on original source code (master)
  • If you are working on a team collaboratively, its a wise idea to branch and work on your branch instead of messing up or messing with master.

checkout

We use git checkout command to switch between branches.

 ~> git checkout branchname
    # If you want to go back to master branch for example.
 ~> git checkout master

clone

git clone command is used for cloning the remote over to your local hard drive.

 ~> git clone https://github.com/dahal/demo

commit

After we add (stage) the changes with git add command, it stages the changes being made to the repo temporarily, git commit adds/confirms those changes.

~> git commit -m "Commit Message"
~> git commit -am "Commit Message"
  # You can skip git add command by using above command.

fetch

git fetch command will help you see the changes between remote repo and local repo. Lets say you have a repo called demo on your github and a cloned copy of that repo on your computer. Somebody made some changes to that repo on github and pushed it over. But you are still working on originally cloned repo, you wanna make sure and merge if there any changes before you push it. 

 # First add the remote branch you are comparing with:-
 ~> git remote add remote_branch https://github.com/dahal/demo.git
 ~> git fetch remote_branch
    remote: Counting objects: 96, done.
    remote: Compressing objects: 100% (69/69), done.
    remote: Total 96 (delta 33), reused 78 (delta 27)
    Unpacking objects: 100% (96/96), done.
    From https://github.com/dahal/demo
        09ae519..a787cba  master     -> remote_branch/master

merge

git merge command is used while working with branches and when you want to merge the changes between two branches. git branch command can also be used while merging/syncing changes from remote to local repo.

 ~> git checkout master
 ~> git merge gh-pages 
    # this will merge changes you have made on gh-pages over to master

 ~> git merge remotebranch/master
    # this will sync changes made on remote branch over to local master

pull

Lets say you are working at your office computer and you made some commits to your repo at your office computer and pushed it over to github. Before you came to office, you were originally working on the same repo at your house. So, now you go back home only to find out your local repo is not up to date. What do you do? Pull it ya'll.

 ~> git pull

push

You have added, committed, merged and everything is good to go. Now you want your code to be pushed (uploaded) on github/bitbucket of any other git hosting sites, what do you do?

 ~> git push -u origin master # if you are pushing for the first time
    # For here and out->
 ~> git push origin master
    # or simply
 ~> git push

reset

Shit happens, you mess up sometimes. You write 100's of lines of code, add them commit them only to realize that you screw up somewhere. At this point, git reset is your best friend.

 ~> git reset

rm

git rm is an interesting command. We generally think rm just removes/deletes the file/folder in unix, that is true. But while combining this command with git, it not just removes it, it adds and commits (with empty message of course)

 ~> cd demo
 ~> touch index.html
 ~> git add .
 ~> git commit -m "Added index file"
    # If you just rm index.html, thank you will need to add, commit again to track the deletion. But, git rm will take care of everything for you.

 ~> git rm index.html
    # So cool.

status

git status is my best friend when working with git/github. If you look track of where you were, like adding/committing etc, just git status, and you will see what is the status of current repo you are working on, it even gives you what you can do next.

~> git status
    On branch master
    Changes to be committed:
    (use "git reset HEAD <file>..." to unstage)

        deleted:    index.html

    # Now you know what to do...
~> git commit -m "Deleted index.html"
    [master 587bd0a] Deleted index
    1 file changed, 1 deletion(-)
    delete mode 100644 index.html
